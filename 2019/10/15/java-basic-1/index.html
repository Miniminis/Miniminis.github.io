자바 복습을 시작한다.

자바 가상 머신(Java Virtual Machine)

자바 언어가 나오기 이전에는 각 운영체제별로 다른 프로그램을 코딩하여 각각 배포하는 시스템이었다고 한다. 그러나 자바 언어의 경우는 아래와 같은 과정으로 이전보다 효율적인 프로그래밍이 가능하게 되었다.

  1. 자바 언어로 프로그래밍 (.java 확장자)
  2. 자바 컴파일러를 통해 자바 바이트 코드로 컴파일 됨(.class 확장자)
  3. 이제 각 운영체제별로 다른 JVM 을 통해 이 바이트 코드를 읽어 프로그램을 실행하게 됨

자바 가상머신은 이처럼 하나의 자바 코드로 동일한 프로그램을 운영체제와 실행환경에 관계없이 실행할 수 있게 해주는 역할을 수행한다.

*** 참고 글쓰는 개발자_JBee님의 블로그
자바 프로그램 실행과정

  1. 자바프로그램이 실행된다. JVM은 OS 로부터 메모리를 할당받는다. 그 메모리는 용도에 따라 여러 영역으로 나뉘어 관리된다.
  2. 자바 코드가 자바 컴파일러에 의해 자바 바이트 코드로 컴파일 된다. (.java –> .class)
  3. Class loader 를 통해 class 파일들을 JVM으로 로딩한다.
  4. 로딩된 class 파일들이 execution engine 을 통해 해석된다.
  5. 해석된 바이트 코드들은 Runtime data areas 에 배치되서 실행됨
  6. 이 사이클을 수행하는 동안 JVM은 thread synchronization 이나 gc 같은 작업을 수행하게 된다.

클래스 로더

자바 클래스 파일들을 로드하고 링크를 통해 배치되도록 하는 역할을 수행한다. 자바는 런타임에(즉 처음으로 클래스가 참조되는 순간에) 참조가 되는 언어인데, 바로 이 클래스 로더가 그 역할을 수행한다.

가비지컬렉터(Galbage Collector)

자바 프로그램을 실행하는 과정에서 낭비되는 자원들이 있는데, 가비지 컬렉터는 이를 자동으로 관리하여 불필요한 메모리의 낭비를 막아준다.
가비지 컬렉터는 아래와 같은 방식으로 실행된다.

  1. 새롭게 생성되는 객체들이 모인 eden 영역에서 일차적으로 낭비되는 자원들을 없엔다.
  2. eden 에서 살아남았을 경우, survivor 영역으로 이동
  3. 1번과 2번의 과정이 반복되고 일정시간이 지나면 오랫동안 참조되어왔다는 의미로 old 영역으로 이동하게 됨
  4. GC는 old 에서 참조되고 있지 않은 모든 객체들을 한꺼번에 삭제하게 된다. 시간이 오래 걸려서 이 작업을 수행하고 있는 스레드 이외의 다른 스레드의 작업들은 모두 중지된다.

가비지 컬렉터가 자원을 관리하는 기준은 힙 내 객체들의 ‘reachability’이다.
여러 참고 관계의 시작인 Root Set of Reference 에서 다른 객체들로의 참조가 시작되고 그 참조는 꼬리에 꼬리를 물게 된다. 이러한 참조 사슬에 얽히지 않고 둥둥 떠있는 객체들을 ‘가비지’ 하고 판단하며 소멸의 대상으로 간주하여 없에버린다.

Java Collection

자바 컬렉션은 다양한 데이터들을 다루는데 표준화된 클래스들을 제공해주어 개발자가 각 자료구조를 직접 구현하지 않고 편하게 사용할 수 있도록 해준다. 또한 자료구조의 경우는 미리 그 크기를 정해놓아햐하는 경우가 있는데, 컬렉션의 경우는 객체의 수를 동적으로 정할 수 있기 때문에 불필요한 메모리의 소모도 예방할 수 있다.

List : ArrayList, LinkedArrayList

  1. 데이터의 순서가 있음
  2. 데이터의 중복 저장을 허용

Map : HashMap, TreeMap

  1. key-value 형태로 저장됨
  2. key는 중복될 수 없고 value는 중복가능
  3. 순서 보장하지 않음 (순서 보장은 LinkedHashMap 이용)

Set : HashSet, TreeSet

  1. 집합
  2. value 중복 불가

Stack & Queue :

  • Stack : 후입선출구조. 가장 나중에 넣은 데이터가 가장 먼저 출력됨
  • Queue : 선입선출구조. 가장 먼저 넣은 데이터가 가장 먼저 출력됨
