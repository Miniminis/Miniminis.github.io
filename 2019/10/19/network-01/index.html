Network / 브라우저에서 서버까지 전체 흐름 정리

2019-10-19

#network

글쓰는 개발자 __jbee 님의 블로그를 보면서 네트워크와 운영체제를 공부하고 있다. 워낙 알기 쉽게 글을 잘 써주셔서 막연하게만 느껴졌던 네트워크가 재미있어지는 것 같다. 지금도 계속 코딩을 계속 하고 있기에 내가 개발하고 있었던 프로그램의 흐름들도 다시 한 번 정리하게 되는 계기가 되고 있다.

재엽님이 여러 페이지에 걸쳐 설명해주신 부분을 한 페이지로 압축해서 한눈에 보고자 한다. (직접 위의 링크를 통해 들어가보면 훨씬 자세한 설명을 볼 수 있다. 나는 개인적인 흐름 정리용으로 본 포스트를 작성한다. )

사용자의 요청이 서버로 전달되는 큰 흐름은 굵은 글씨로 적힌 1 - 16번과 같다.


웹 브라우저, 웹 클라이언트

사용자의 요청을 받아 서버로 전달. 즉 요청하는 쪽.
택배 송신자의 개념

1. 사용자가 요청 : 버튼을 클릭 or 주소창에 url 직접 입력 등

2. 클라이언트는 사용자의 요청을 요청메시지(http request)로 만들어서 운영체제의 프로토콜 스택으로 토스

  • 클라이언트는 서버로 바로 사용자의 요청을 전달할 수 있는 능력이 없기 때문에, 사용자의 요청을 프로토콜 스택에 넘기는 정도로만 역할을 수행함

  • URL :

    • http나 ftp 등 서버에 엑세스 하는 방법을 규정해놓은 프로토콜을 따라 작성되어야함.
    • // 이후 : 요청하는 서버의 주소
    • /{번호 or 문자 등} : 데이터 출처의 경로명
    • Http 프로토콜 :
    • 클라이언트와 서버가 메시지를 주고받는 내용이나 순서에 대한 약속
    • 담기는 내용 : 1) 무엇을 2) 어떻게 3) 추가정보
      • 1) 무엇을 : URI (Uniform Resource Identifier). 액세스 대상.
      • 2) 어떻게 : Http method
      • 3) 추가정보 : header 파일을 이용.

  • Http Request Message 의 구조 or 형식

  • Http Response Message의 구조 or 형식


  • DNS Resolver(DNS 클라이언트라고 이해함)

    • 요청메세지를 보내고자 하는 서버의 IP주소를 의뢰하는 쪽
    • OS 내부에 존재하는 Socket 라이브러리에 있음
    • 전달 메시지는 보통의 경우 다음과 같음
      • 이름 : to which server
      • 클래스 : IN(internet)
      • 타입 : A(Address : IP)

  • DNS (Domain Name System) (DNS 서버라고 이해함)

    • 컴퓨터들은 허브-서브넷-라우터를 통해 네트워크 망을 형성. 각 서브넷 컴퓨터가 가지는 고유한 주소가 바로 IP 주소 (허브로 연결된 호스트주소+서브넷의 주소인 네트워크주소가 결합된 형태임)
    • 각 웹사이트들은 IP 주소가 아니라 사용자에게 기억되기 쉬운 도메인 주소를 따로 설정함. 이 도메인 주소로 요청이 들어오면 DNS resolver - Protocol stack을 통해서 DNS 서버로 이동하여 DNS 서버 내에 정리된 일종의 주소록을 통해 요청의 목적지가 되는 서버의 IP 주소를 획득할 수 있게 되는 것!

  • DNS 서버가 가지고 있는 주소록, 즉, 리소스 테이블에 맞는 IP 주소가 없을 경우?

    • 전세계의 DNS 서버가 연대하여 그에 맞는 IP 주소를 찾아줌 (13개의 서버 존재)


프로토콜 스택 (Protocol Stack)

택배회사의 개념
운영체제에 내장되어있는 네트워크 제어용 소프트웨어라고 함.

3. 클라이언트에서 받은 사용자의 요청 메시지를 패킷안에 포장

4. 패킷에 서버 주소(받는 사람) 등의 제어정보를 덧붙임

5. 이 패킷(디지털 정보상태)을 LAN 어뎁터에 전달

6. LAN 어뎁터는 해당 패킷을 컴퓨터와 인터넷이 알아들을 수 있는 전기/빛 신호로 변환시킴

7. LAN 케이블로 이 메시지를 송출시킴

프로토콜 스택은 3가지부분으로 이루어져 있음.
이 내부에서 UDP 통신으로 DNS 서버 통신 - TCP 에서 요청정보 패킷으로 포장 및 IP로 전달 - IP에서 LAN 어뎁터 및 케이블을 통해 패킷 송수신 등의 과정을 거쳐 요청 메시지가 전달됨.

1) TCP (Transmission Control Protocol) :

  • 데이터의 송수신 담당
  • 연결형. 소켓으로 클라이언트와 서버를 소켓으로 연결한 후 통신. 즉, 소켓과 소켓 사이를 연결한 후 데이터 송수신. 종단점-종단점 통신.
  • 주로 어플리케이션의 데이터 송수신을 담당
  • 클라이언트와 서버가 서로 메시지를 받았는지를 주기적으로 확인할 수 있으므로 신뢰성이 높고 메시지가 보낸 순서대로 전달되어 순차적이다.
  • 바이트 스트림 단위(세그먼트)로 데이터가 전달됨

2) UDP (User Datagram Protocol) :

  • 데이터의 송수신 담당
  • 비연결형. 클라이언트와 서버가 연결되어있지 않고 통신.
  • 주로 DNS 서버와의 짧은 제어용 데이터 송수신에 이용됨
  • 사전설정할 내용이 없으므로 짧고 빠른 응답을 할 수 있음.
  • 하지만 연결이 되지 않은 상태로 메시지를 전달하므로, 데이터의 전달, 지연, 손실에 대해 전혀 보장할 수 없다.
  • 실시간 멀티미디어 RTP(Realtime Transport Protocol) 의 기반이 됨
    RTP : 오디오, 비디오 등을 실시간 통신. UDP 패킷을 여러 개 사용하는 멀티플랙싱 방식

3) IP :

  • 패킷의 송수신 담당
  • LAN Driver 존재
  • ICMP : 패킷 송수신 시 발생할 수 있는 오류, 이슈들을 통지하는 역할
  • ARP : IP주소에 대응하는 Ethernet의 Mac 주소를 조회하는 역할 담당

Socket Library

  • 한번 접속한 DNS address를 기억함. 대신 유효기간 존재.
  • 프로토콜 스택 내부에 존재하는 메모리 공간을 Socket 이라고 함. 프로토콜 스택은 주기적으로 이 소켓을 참조하며 통신하게 됨
  • 통신제어용 제어정보를 이 소켓 영역에 기록하게 됨
  • 예) IP 주소, Port 번호, 통신진행상태, 데이터기록 등
  • 모든 소켓에는 describtor가 존재. 클라이언트나 서버에 존재하는 여러 소켓들을 구분하기 위해서.


허브, 라우터

8. LAN 케이블을 따라서 패킷이 허브로 전달되고 허브는 다시 라우터로 패킷을 전달한다. 토스토스!!!!

9. 라우터는 패킷을 프로바이더로 다시 토스

10. 이제 드디어 인터넷으로 들어가게 됨!!!!



방화벽, 캐시서버

11. 패킷은 인터넷을 통해서 드디어 웹서버 측의 LAN에 도착하게 됨!

12. 이때, 웹 서버를 지키는 방화벽이 해당 패킷을 검사함

  • 이 정보가 예전에 요청됐었거나 웹 서버까지 갈필요없이 캐시서버에 이미 저장되어있는 내용인지 확인

    13. 캐시서버에 요청 내용이 없고 웹서버까지 갈 필요가 있다고 판단되면, 방화벽이 문을 열러서 웹서버로 전달함



웹 서버

택배 수신자의 개념

14. 패킷은 드디어 웹서버에 도착. 웹 서버 내부에 존재하는 프로토콜 스택이 해당 패킷을 추출해서 다시 전기 메시지를 디지털 메시지 형태로 변환하는 등 메시지를 복원하고 웹 서버 어플리케이션에 해당 내용을 남긴다.

15. 웹 서버는 요청 메시지를 분석하고 이에 맞는 응답 메시지를 준비

16. 다시 같은 과정을 거쳐서 클라이언트로 전달된다.
